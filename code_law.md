# Code Law 编程法则

> 源自 Andrej Karpathy 的 LLM 编码实践总结

## 核心原则

### 1. 编码前思考 (Think Before Code)

**不要假设。不要隐藏困惑。呈现权衡。**

- **明确说明假设** — 如果不确定，询问而不是猜测
- **呈现多种解释** — 当存在歧义时，不要默默选择
- **适时提出异议** — 如果存在更简单的方法，说出来
- **困惑时停下来** — 指出不清楚的地方并要求澄清

### 2. 简洁优先 (Simplicity First)

**用最少的代码解决问题。不要过度推测。**

- 不要添加要求之外的功能
- 不要为一次性代码创建抽象
- 不要添加未要求的"灵活性"或"可配置性"
- 不要为不可能发生的场景做错误处理
- 如果 200 行代码可以写成 50 行，重写它

**检验标准：** 资深工程师会觉得这过于复杂吗？如果是，简化。

### 3. 精准修改 (Precise Changes)

**只碰必须碰的。只清理自己造成的混乱。**

- 不要"改进"相邻的代码、注释或格式
- 不要重构没坏的东西
- 匹配现有风格，即使你更倾向于不同的写法
- 如果注意到无关的死代码，提一下 —— 不要删除它

**检验标准：** 每一行修改都应该能直接追溯到用户的请求。

### 4. 目标驱动执行 (Goal-Driven Execution)

**定义成功标准。循环验证直到达成。**

将指令式任务转化为可验证的目标：

| 不要这样做... | 转化为... |
|--------------|-----------------|
| "添加验证" | "为无效输入编写测试，然后让它们通过" |
| "修复 bug" | "编写重现 bug 的测试，然后让它通过" |
| "重构 X" | "确保重构前后测试都能通过" |

对于多步骤任务，说明一个简短的计划：

```
1. [步骤] → 验证: [检查]
2. [步骤] → 验证: [检查]
3. [步骤] → 验证: [检查]
```

## 实践检查清单

### 编码前
- [ ] 是否理解了用户的真实需求？
- [ ] 是否存在多种实现方式？是否呈现了权衡？
- [ ] 是否有不确定的地方需要澄清？

### 编码中
- [ ] 是否只修改了必要的代码？
- [ ] 是否保持了代码简洁，没有过度设计？
- [ ] 是否遵循了项目的现有风格？

### 编码后
- [ ] 是否验证了修改达到了预期目标？
- [ ] 是否清理了自己引入的死代码？
- [ ] 是否运行了测试确保没有破坏现有功能？

## 警示信号

如果出现以下情况，说明可能违反了法则：

- ❌ 修改了大量与需求无关的代码
- ❌ 引入了复杂的抽象层
- ❌ 添加了用户没有要求的功能
- ❌ 没有验证就假设实现正确
- ❌ 对不清楚的需求做出了假设

## 核心洞察

> "LLM 非常擅长循环执行直到达成特定目标……不要告诉它该做什么，给它成功标准，然后看着它完成。"

**目标 > 指令**：将指令式指令转化为带有验证循环的声明式目标。
